# Performance Measurement Report: Diff Viewer Optimization (Large Scale)

## 개요

`DiffViewer` 컴포넌트 내의 코드 diff 계산 로직(`getDiffFile`)에 `useMemo`를
적용한 결과에 대한 최종 성능 측정 보고서입니다. 테스트를 위해 5,000줄 규모의
대형 코드 데이터를 사용하여 최적화 전후의 차이를 극대화하여 측정했습니다.

## 측정 방법

- **대상**: `src/components/pullRequest/fileChanges/diffViewer/index.tsx` 내의
  `getDiffFile()` 연산.
- **데이터**: 약 5,000줄의 자동 생성된 코드 (기본 라인 및 수정 라인 포함).
- **도구**: 브라우저 개발자 도구 콘솔 및 `performance.now()`.
- **시나리오**:
  1. **Initial Mount (최초 렌더링)**: 아코디언을 펼쳐 파일 내용을 처음으로
     로드할 때.
  2. **Re-render (리렌더링)**: 창 크기 조절(Resize) 등 상태 변화로 인해
     컴포넌트가 다시 그려질 때.
  3. **Remount (재마운트)**: 아코디언을 닫았다가 다시 열어 컴포넌트가 다시
     생성될 때.

## 측정 결과

| 시나리오          | Diff Calculation (순수 연산) | Diff Operation (총 소요 시간) | 비고                              |
| :---------------- | :--------------------------- | :---------------------------- | :-------------------------------- |
| **Initial Mount** | **약 114ms ~ 128ms**         | **약 115ms ~ 129ms**          | 5,000줄에 대한 Diff 알고리즘 수행 |
| **Re-render**     | **0.00 ms (Skipped)**        | **0.00 ms**                   | `useMemo` 캐시 적중으로 연산 생략 |
| **Remount**       | **약 56ms ~ 70ms**           | **약 57ms ~ 71ms**            | 컴포넌트 재생성으로 인한 재계산   |

## 분석 및 결론

1.  **메인 스레드 블로킹 방지**: 5,000줄 규모의 파일에서 발생하는 **120ms**
    수준의 연산은 브라우저의 메인 스레드를 일시적으로 멈추게 하여 사용자에게
    "프리징(Freezing)" 현상을 느끼게 할 수 있습니다.
2.  **연산 비용 완전 제거**: `useMemo`를 통해 리렌더링 시 이 120ms의 연산 비용을
    **0ms**로 만들었습니다. 이는 댓글 작성, 탭 전환 등의 UI 상호작용 시 지연
    없는 쾌적한 UX를 보장합니다.
3.  **Remount 시 성능 차이**: 최초 로드(120ms)보다 재마운트 시(60ms) 연산이
    빨라진 것은 브라우저 엔진(V8 등)의 JIT 최적화 효과로 해석됩니다.
4.  **최종 결론**: 코드 리뷰 도구와 같이 텍스트 연산량이 많은 애플리케이션에서
    `useMemo`를 통한 전략적 캐싱은 UI 반응성 유지에 필수적인 최적화 기법임을
    실측 데이터를 통해 증명했습니다.
